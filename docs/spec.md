# ノードグラフ型 プロンプト生成・組み合わせ描画アプリ 仕様書

## 1. 概要

ノードグラフ上で Stable Diffusion 用のプロンプトを意味単位で組み立て、**組み合わせ（直積）で複数枚を一括生成**するデスクトップアプリ。

- キャラクター・背景・アクション・カメラなどを「意味ノード」として設定する。
- キャラ同士の動作パターン、カメラパターン、シード値などを掛け合わせて複数枚を生成する。
- メタデータにプロンプトを持つ既存生成画像を「参照ノード」として読み込み、**キャラだけ／背景だけを差し替えた**バージョンを生成できる。
- 描画バックエンドは **WebUI Forge**（A1111 互換 API）。**Forge Couple／領域分割系は使わない**。

## 2. アーキテクチャ

### 2.1 二層構成

```
[意味グラフ（本アプリ）]  →  プロンプトにコンパイル  →  [WebUI Forge API で実行]
```

低レベルなサンプラ等は Forge に任せ、本アプリは「意味ノードの編集」と「プロンプトへのコンパイル」「ジョブ発行」を担当する。

### 2.2 想定スタック

- フロント: Electron + React（React Flow）+ TypeScript + Vite
- バックエンド: Python / FastAPI
- 永続化: SQLite（グラフ状態・生成画像メタデータ・系譜）
- 補助 LLM: ローカル LLM（llama.cpp）— 可視性フィルタ判定・参照プロンプト分解に使用

### 2.3 描画バックエンド（Forge）

- 生成: `POST /sdapi/v1/txt2img`
- 参照画像メタデータ抽出: `POST /sdapi/v1/png-info`
- モデル一覧等: `GET /sdapi/v1/sd-models` ほか
- メタデータ形式は A1111 互換（positive → `Negative prompt:` → `Steps/Sampler/Seed…` の設定行）。
- 領域分割（Forge Couple / Regional Prompter）は不使用。複数キャラは単一プロンプトで扱う（§6）。

## 3. 設計の中核原則

仕様全体を貫く3つの原則。

### 3.1 要素を2階層に分類する

各要素を「空間に紐づくか／全体に効くか」で分け、これを**可視性フィルタの対象か否か**に対応させる。

| 区分 | 性質 | 可視性フィルタ | 該当ノード |
| --- | --- | --- | --- |
| 空間に紐づく | 画角で写ったり消えたりする | 触る（除去対象） | character / solo action / background |
| 全体に効く | 画角に関係なく全体に効く | 触らない（素通り） | interaction / lighting / style |

カメラがこのフィルタを駆動する。

### 3.2 ノードの境界 ＝ 直積の軸

- 1ノードに**束ねる**＝連動（固定された組として扱う）。
- ノードを**分ける**＝直積（独立に掛け合わせる）。

「束ねるか分けるか」だけで、交差させるかどうかを制御する。

### 3.3 鎖の形で束縛を明示する

`girl → girl action → scene` のように、キャラとそのソロアクションを鎖でつなぐことで「このアクションはこのキャラのもの」という束縛を構造的に表す。Scene は上流をたどるだけで「キャラ＋アクション」を1つの束として拾える。

## 4. ノード一覧

### 4.1 Character ノード

- 役割: キャラの外見プロンプト断片。
- 内部はカテゴリ分け（顔・表情 / 髪 / 上半身 / 下半身 / 全身 / 小物）。
- 各カテゴリに空間ゾーンタグ（head / upper / lower / feet）を持たせ、可視性フィルタが参照する。
- 出力: character ハンドル。
- 多値（バリアント集合）にでき、組み合わせ生成のスイープ軸になる。

### 4.2 Solo Action ノード

- 役割: キャラの鎖の中で、ソロのポーズ・動作を「パススルー加算」する。
- 入力: Character を1つ。出力: キャラ＋アクションの束。
- 対象例: `sitting` / `arms crossed` / `waving`。
- 可視性フィルタの対象（空間に紐づく）。
- 鎖: `character → solo action → scene`。

### 4.3 Interaction ノード

- 役割: 2人にまたがる相互作用を表す**独立ノード**。
- Character 入力を持たず、Scene の共通側に直接つながる。
- 対象例: `hugging` / `holding hands` / `looking at each other` / `back-to-back`。
- 可視性フィルタは触らない（全体に効く）。
- ソロ action とはハンドル型を分け、挿し間違いを構造的に防ぐ（`Act` ピン / `Int` ピン）。
- 補足: 領域系を使わない本構成では、視線方向などの補助は各キャラの solo action 側、関係の本体は Interaction 側に置くと破綻しにくい。

### 4.4 Background ノード

- 役割: 背景・環境のプロンプト断片。
- 各要素に空間ゾーンタグ（floor / ceiling / wall / far / near）を持たせる。
- 可視性フィルタの対象（例: `from below` なら `glowing floor` を落とす）。
- 照明語は持たせない（Lighting に寄せる）。
- Scene の共通側につながる。

### 4.5 Lighting ノード

- 役割: 全体照明（`neon` / `dim` / `spotlight` / `rim light` 等）。
- Background とは**別ノード**。可視性フィルタの対象外（全体に効く）。
- 多シーンへ再利用しやすく、スイープ軸にもなる。

### 4.6 Camera ノード

- 役割: フレーミング＋アングルの**プリセット**を保持する。
- 入力規則: **1行＝1プリセット、行内カンマ＝束**（例: `head focus, from above`）。
- フレーミングとアングルは相関するためキュレーションした束にする（全直積にしない）。
- プリセット内のアングルは可視性フィルタが読む情報になる。
- 多値（スイープ軸）。

### 4.7 Style ノード

- 役割: 全体のスタイル・品質タグ。可視性フィルタ対象外。スイープ軸。

### 4.8 Seed ノード

- 役割: シード値。単一または複数（スイープ軸）。

### 4.9 Reference ノード

- 役割: メタデータ（プロンプト）を持つ既存画像を読み込み、差し替えの起点にする。
- `png-info` で取り出したプロンプトをローカル LLM で `character / background / action / camera / style` のバケツに**分解**する。
- バケツは編集可能（LLM が提案、ユーザーが修正）。
- カテゴリごとの出力ハンドルを持ち、Scene の各スロットへ既定値を供給する。
- Scene のスロットに専用ノードを挿すと、そのカテゴリだけ上書きされる（§8）。

### 4.10 Scene ノード

- 役割: 合成器。上流をたどってプロンプトを組み立てる。
- 入力: キャラの鎖（複数）＋ interaction ＋ background ＋ lighting ＋ style（＋ Reference 由来の既定値）。
- 処理: スロット解決 → 可視性フィルタ → 順序・weight 付け → 最終 positive / negative プロンプト生成。
- 複数キャラ時は単一プロンプトに合成（§6）。

### 4.11 可視性フィルタ（Scene コンパイル内のステップ）

- 役割: 「画面に写らない要素を落とす」。キャラ・背景を同じ1ステップで処理する。
- カメラのアングルから可視ゾーンを判定し、該当する空間ゾーンタグの要素を除去する。
- ローカル LLM 駆動。`(camera, 対象内容)` のハッシュで**キャッシュ**し、再現性を確保。
- 「実際に除去されたトークン」を表示して**手修正可能**にする。
- lighting / style は素通り。

### 4.12 Batch ノード

- 役割: 全スイープ軸の直積を計算し、ジョブを発行する。
- 展開モード: 全列挙（combinatorial）/ ランダム抽出（重み付き可）/ 固定（pin）。
- 軸の連動が必要な場合は zip（対応づけ）モード、または該当軸の束ね（プリセット化）で対応。
- **ドライラン**: 生成前に総枚数とサンプルプロンプト数件を提示してから確定（直積の爆発に対する安全弁）。
- 系譜を SQLite に記録。

## 5. プロンプト合成パイプライン（Scene コンパイル順）

1. スロット解決（Reference 既定値 ＋ 専用ノードによる上書き）。
2. 収集: 各キャラの鎖（character ＋ solo action）、interaction、background、lighting、style。
3. 可視性フィルタを空間要素（キャラのゾーン、背景のゾーン）に適用。キャッシュ＋手修正。
4. 複数キャラ時は単一プロンプトに合成（§6）。
5. 順序づけ＋ weight 付与。negative プロンプトを組み立て。
6. seed / 設定を決定。
7. Forge `txt2img` へ送信。

## 6. 複数キャラクターの扱い（領域系なし）

- 対象は最大2人（3人以上は現状スコープ外）。
- 単一プロンプトで合成する。グラフ構造（girl/guy の鎖、interaction の共通配置）は変えない。
- 緩和策（プロンプトのみ）:
  - 人数タグ（`2girls` / `1girl, 1boy` 等）。
  - 位置ヒント（`on the left` / `on the right`）。
  - `BREAK` でキャラブロックを CLIP チャンク分割し、特徴の混ざりを軽減。
  - interaction・background・lighting・style は共通部分に置く。
- 既知の限界: 領域分割なしのプロンプトのみでは、2キャラ間の特徴の混ざり（色移り等）を完全には防げない。上記は軽減策であって除去ではない（§11、§12）。

## 7. 組み合わせ生成モデル

- **ノード内**: バリアント集合（列挙される選択肢）。各バリアントは単一タグまたは束（タグの AND、連動）。
- **ノード間**: 直積（ノード境界＝軸）。
- 展開モード: 全列挙 / ランダム抽出（重み付き）/ 固定。
- 連動が要る場合（例: ローアングルのときだけポーズをしゃがみに）は zip、または束ね。
- ドライランで総枚数・サンプルを確認してから実行。

## 8. 参照画像とスロット上書き

- Reference ノードは全カテゴリのスロット既定値を供給する箱として働く。
- Scene の各カテゴリスロットは入力を1つ取り、専用ノードを挿したスロットだけが勝つ（レイヤード・オーバーライド）。
- 「キャラ差し替え」＝ character スロットのみ新ノードに置換、残りは参照流用。
- 「背景差し替え」＝ その逆。
- 注意: プロンプトのみの差し替えでは「背景を維持」＝**背景の記述を維持**であり、ピクセル単位の一致ではない。画像全体を再生成するため構図は動く。
- 2キャラの参照プロンプトを2人ぶんに自動分解するのは難度が高く、手修正前提とする。

## 9. データと再現性

- SQLite に保存: グラフ状態 / 生成画像＋メタデータ / 系譜（どのグラフ状態がどの画像を生んだか）。
- LLM の判定（可視性フィルタ・参照分解）はキャッシュし、編集可能にする。
- 「あの設定でもう一度」を成立させることを再現性の目標とする。

## 10. スコープと既知の制約

- キャラは最大2人（3人以上は除外）。
- 差し替えはプロンプトのみ（記述レベル。構図はドリフトする）。
- 領域系を使わないため、2キャラの特徴混ざりは緩和のみで完全除去ではない。
- LLM 判定は非決定的なので、キャッシュ＋手修正で再現性と精度を担保する。

## 11. 今後の拡張余地（グラフ構造を変えずに差し込める）

- 2キャラ分離の強化: 領域条件づけ（Forge Couple 等）、inpaint、ControlNet（OpenPose/DWPose/depth）への切り替えは、Scene のコンパイル先を差し替えるだけで対応可能。
- キャラの一貫性: プロンプトのみでは弱い。必要時はキャラごとの LoRA、IPAdapter＋参照画像、textual inversion を検討。
- 背景のピクセル維持が必要になった場合は inpaint / img2img へ。

### 付録: ノード接続の俯瞰

```
girl → girl action ─┐
guy  → guy action  ─┤
       interaction ─┤
        background ─┼→ scene node → (可視性フィルタ) → Forge txt2img → 画像
          lighting ─┤        ↑
             style ─┘     camera が駆動

camera / character / seed / … （多値ノード）→ Batch ノード（直積・ドライラン）→ ジョブ群
```
